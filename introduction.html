Introduction

Technically speaking, in computing a "dot file" is a hidden directory or file on a UNIX/Linux system that are so named because they are prefixed with a full stop. Dot files within a user's home directory contain most of the user-specific configuration available on a desktop and are configured manually or through the use of tools.

While not all strictly dot files, the files contained here are some of the customization I regularly use on my Linux machines as well as on the Linux machines of those whom I convert from the lesser operating systems. They are offered here with plentiful documentation in the hopes that they will be useful to others, whether as an inspiration to explore some different areas of Linux, as a shortcut to get a Linux machine up and running, or as a guide for users not familiar with how to configure some of these applications (like XMonad).

Overview

There are two directories at the top level of the tree: my $HOME folder and the /etc directory. /etc contains the system-wide configuration files I often install on a new machine or upon daemon or application installation. There isn't much in terms of surprise or intrigue in these files as they are relatively standard. The magic is in the $HOME folder.

The $HOME folder is divided into two top-level folders and a file. The file is .bashrc, the main entry-point to user-specific Bash configuration. This file loads the files within the .bash directory. These files control aliases, helper functions, terminal prompts, executable paths, and much more.

The second folder is .xmonad with contains all configuration for the xmonad window manager. xmonad is a dynamically-tiling window manager for X11 written in Haskell. xmonad is very speedy and lightweight because its configuration is also written in Haskell and compiled prior to running (it takes only two or three seconds). xmonad can also be compiled on-the-fly while its running so you never have to leave xmonad to change your configuration.

The Bash and xmonad configurations are discussed in more detail below.

Bash Configuration

Fortunately, Bash configuration is fairly straitforward. On most Linux systems, $HOME/.bashrc is the standard configuration file for non-login, interactive shells. That is, this configuration is loaded when you open a virtual console on one your TTYs or a terminal emulator in your desktop environment, but it is not loaded when a script is executed.

When exploring my configuration, begin with .bashrc. The documentation is quite detailed.

To use these, simply copy these dot files into your home folder (being sure to rename to .bashrc and .bash/) and re-log-in. The dot files can be downloaded by clicking the "View on GitHub" link at top right and there are more detailed instructions there for installation.

XMonad Configuration

xmonad is more difficult to set up than Bash, unfortunately. xmonad is a wonderful tiling window manager (my personal favorite) but too many people eschew it because the configuration is complicated and, if you're not a Haskell wizard, not well documented. I am no Haskell wizard, so my configuration is based on many hours of troubleshooting, guess-and-check, API documentation browsing, and gleaning what I can from the configurations of others. I am sure there are improvements to be made, more efficient ways of doing things, and better approaches - but mine works well enough.

For a user to run xmonad, she must have a properly-created $HOME/.xmonad/xmonad.hs that initializes xmonad and creates a basic setup. There is no useful default environment. Once the file is created, it must be compiled by running the following command as your user - not as root nor with sudo. 

$ xmonad --recompile

If all is well, the command will exit with no complaints. Otherwise, the Haskell compiler (GHC) will provide an error that can be difficult to diagnose without being familiar with Haskell. If you run into one of these errors, I wish you luck. I fixed mine through trial and error and became familiar with Haskell largely through experimenting with xmonad.

The compiler outputs an executable binary file in the $HOME/.xmonad directory named something like xmonad-x86_64-linux that the xmonad command uses to run your environment. You will also find several other files in that directory that are generated and populated during runtime. We don't need to concern ourselves with those here.

The xmonad.hs file contains very detailed documentation about exactly what it does, so we won't get into that here. It does, however, reference the files in $HOME/.xmonad/bin so that directory must be copied along with xmonad.hs into the $HOME/.xmonad directory. See the repository of dot-files on GitHub (link at upper right) for more information on installation.
